Case Study · 04

PayFlow v2

Money can never just appear or disappear. That's the one rule this whole project is built around.

RoleSolo Developer
StackGo · Scala · Ruby
StatusDeployed, -race Tested
The Problem

Most "payment system" projects are basically a charges table with a REST wrapper around it. That's how this one started too — until a real bug turned up: a race condition that let the same idempotency key create two charges instead of one. Everything after that point was built to make sure that kind of failure gets caught by a test, not discovered in production.

The Approach

Three languages, each doing the job it's actually good at — not picked to show off:

  • Go runs the core service, since payment code lives or dies by how it behaves under concurrency
  • Scala runs the risk engine, where pattern matching over case classes keeps growing rule sets from turning into spaghetti
  • Ruby runs the CLI, so auditing charges by hand is pleasant instead of reading raw JSON
  • Double-entry ledger bookkeeping — every charge posts a debit and an equal credit, so the whole ledger has a mechanical invariant to check: it must always sum to zero
  • Write-ahead logging, fsync'd before a request returns, with a ReplayFromLog function and a /debug/verify-replay endpoint that rebuilds state from the log and checks it against what's live
  • A fail-safe risk engine — if the Scala service is unreachable or errors out, charges fall back to pending_review instead of silently succeeding or crashing
  • Stripe-style Idempotency-Key header support, bearer-token auth, per-caller token-bucket rate limiting, and HMAC-SHA256-signed webhooks
The Bug That Actually Matters Here

The first version checked whether an idempotency key had been seen before under a read lock, released it, then wrote the new charge under a separate write lock — a textbook check-then-act race. Every test passed, because every test up to that point ran one request at a time, and a single-threaded test can't prove anything about two requests landing in the same microsecond.

So the fix was preceded by a test built specifically to break it: 50 identical requests, same idempotency key, fired genuinely concurrently with goroutines and a WaitGroup. First run, it failed immediately — different callers got back different charge IDs for what should have been one logical charge. The real fix collapsed the check and the write into a single critical section, with the event-log append happening inside that same lock so the log and in-memory state can never disagree about what happened first.

Outcome & Honest Limits

The concurrency and ledger story is solid: go test ./... -race passes, and it's literally what would have caught the original bug before a load test found it. What's not fully wired up yet: the live deployment's Docker image doesn't build the Scala risk-engine jar or bundle a JRE, so large charges currently fail safe to pending_review rather than getting a real risk score — a known, documented gap rather than a hidden one. Webhooks are implemented and tested but not enabled live, since there's no receiver configured yet.
